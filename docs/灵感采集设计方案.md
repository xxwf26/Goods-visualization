# 灵感库·小红书最新灵感自动采集 — 设计方案

> 状态：**方案评审阶段（未实现）** ｜ 起草 2026-07-22
> 决策已定：关键词搜索（接受 Playwright 重依赖）＋ 独立候选表 ＋ 采到先进候选队列人工复核

---

## 1. 背景与目标

灵感库现状（已核查）：
- `inspiration` 表字段齐全（source_url / cover_image / images / author / post_tags / categories 等）。
- 已有 `MetaFetcher`（纯 Node http 解析小红书 SSR `__INITIAL_STATE__`，**免 cookie、免浏览器**）＋ `autofillFromUrl`：**贴单个链接即可自动抓取预填**。
- 已有 `AiAnalyzer` 异步分析、图片下载落盘、敏感源过滤。
- **缺口**：只能手动逐条贴链接；无候选/审核队列，灵感录入即可见。

目标：把「手动贴单链接」升级为「**输入采购关键词 → 批量拉小红书最新帖 → AI 预筛 → 进候选队列 → 人工复核转正进灵感库**」。

典型用法：营销采购岗输入「亚克力吧唧」「醒型POP」「谷子周边」等关键词，系统定期/手动拉回最新帖，复核后沉淀成灵感。

---

## 2. 关键技术约束（务必先理解）

小红书有**两条互斥的抓取路径**，这决定了成本：

| 能力 | 技术 | cookie | 浏览器 | 周边现状 |
|------|------|--------|--------|----------|
| 贴链接 / 抓主页 / 抓单帖 | 纯 Node http 解析 SSR | 不需要 | 不需要 | ✅ 已有(MetaFetcher) |
| **关键词搜索**（本方案核心） | **Playwright 拦搜索接口** | **必须** | **必须 Chromium** | ❌ 需新建 |

**结论**：关键词搜索页是 JS 渲染 + 需登录态，现有纯 http 方案打不到。必须引入 Playwright + Chromium(~150MB) + 扫码登录取 cookie，且 cookie 会过期需定期人工重扫。此成本已确认接受。

参考实现：style-atlas（`C:\Users\xiaoxiaocha\tags`）的 `server/src/scripts/xhs-search.mjs`（Playwright 子进程拦 `/api/sns/web/v2/search/notes`）＋ `xhs-login.mts`（扫码登录存 storageState）已验证可用，直接移植。

## 3. 数据模型

### 3.1 新建候选表 `inspiration_candidate`

设计原则（借鉴 ACG 雷达成熟模式）：**候选表 = inspiration 业务字段镜像 + 一组复核脚手架**。转正时业务字段迁入 `inspiration`，脚手架丢弃，候选记录保留（只翻状态，留审计轨迹）。

```sql
CREATE TABLE inspiration_candidate (
  id            BIGINT AUTO_INCREMENT PRIMARY KEY,
  -- 溯源脚手架
  crawl_run_id  BIGINT,                          -- 归属哪次抓取批次
  keyword       VARCHAR(100),                    -- 由哪个关键词采到
  source_platform VARCHAR(50) DEFAULT '小红书',
  -- 业务字段（对齐 inspiration，转正时迁移）
  source_url    VARCHAR(1000) NOT NULL,          -- 帖子链接（含 xsec_token）
  title         VARCHAR(500),
  author        VARCHAR(200),
  author_url    VARCHAR(1000),
  cover_image   VARCHAR(500),                    -- 已下载到本地的封面
  images        TEXT,                            -- 逗号分隔本地图路径
  post_tags     VARCHAR(500),                    -- 帖子话题标签
  description   TEXT,                            -- 正文
  like_count    INT, collect_count INT, comment_count INT,
  -- AI 预筛脚手架
  ai_score      INT,                             -- 0-100 相关度/质量分
  ai_reason     VARCHAR(500),                    -- 一句话理由
  ai_category   VARCHAR(50),                     -- AI 猜的 packaging/peripheral/effect/production
  -- 去重命中
  dedup_inspiration_id BIGINT,                   -- 疑似已存在的灵感 id（软提示）
  -- 状态机
  status        VARCHAR(20) DEFAULT 'pending',   -- pending|adopted|rejected
  reviewed_by   BIGINT,                          -- 谁复核的
  promoted_id   BIGINT,                          -- 转正后生成的 inspiration.id
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_status (status),
  INDEX idx_run (crawl_run_id),
  UNIQUE KEY uq_source_url (source_url(255))     -- URL 去重
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 3.2 新建抓取批次表 `crawl_run`

```sql
CREATE TABLE crawl_run (
  id          BIGINT AUTO_INCREMENT PRIMARY KEY,
  keywords    VARCHAR(500),                      -- 本次搜的关键词（逗号分隔）
  status      VARCHAR(20) DEFAULT 'running',     -- running|ok|failed
  recalled    INT DEFAULT 0,                     -- 召回总数
  new_count   INT DEFAULT 0,                     -- 去重后新增候选数
  error       VARCHAR(500),
  created_by  BIGINT,
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  finished_at DATETIME
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

> **迁移脚本务必写成 `database/fix_schema_vXX_*.sql`**（沿用项目惯例，MySQL8 幂等写法 information_schema + PREPARE），且**入 git**。教训见「升级指南」：xxwf 那边 schema 手工演进不留迁移文件导致同步不全。

## 4. 状态流转

```
[关键词搜索抓取] --AI预筛+去重--> pending --人工/批量--> adopted  (INSERT inspiration, 候选标 adopted + promoted_id)
                                    |                --> rejected (软丢弃, 只翻状态)
                                    └----- restore <----- rejected  (误丢找回)
```

- **一入口两出口 + 可逆**：pending 进；adopted（转正建灵感）/ rejected（软删）出；rejected 可 restore 回 pending；adopted 不可逆（已生成正式灵感）。
- **转正用事务**：`INSERT inspiration` ＋ `UPDATE candidate SET status='adopted', promoted_id=?` 原子完成，避免半途状态不一致。
- **转正前可改字段**：复核弹窗允许改标题/分类/标签再转正（复用现有 `InspirationFormDialog`）。

---

## 5. 去重（三层，键=source_url）

周边灵感天然有稳定 URL，比雷达按名字去重更可靠：
1. **vs 已有灵感**：查 `inspiration.source_url`，命中则给候选打 `dedup_inspiration_id`（**软提示**，不跳过，让人工判断是否重复）。
2. **vs 历史候选**：`inspiration_candidate` 已有该 url（任何状态）→ **硬跳过**（`uq_source_url` 唯一键兜底）。
3. **本批内**：Set 去重。

## 6. AI 预筛（减负，复用现有 AiAnalyzer）

爬到每条先过一遍 AI：
- 判断「是否周边采购相关」+ 给 0-100 分 + 一句话理由 + 猜分类（4 类 packaging/peripheral/effect/production）。
- 打分口径由一段可配置 prompt 驱动（可存现有 `system_setting` KV 表，营销岗可维护「什么样的周边算好灵感」）。
- 候选列表**按 ai_score 降序**，高分浮顶。
- **批量操作**：按阈值「批量转正 ≥85 分」「批量丢弃 <50 分」，按钮显示命中数 + 二次确认（照搬雷达交互）。
- AI 可降级：无 key / 故障时中性放行（score=null），仍进候选待人工复核，不阻断。

## 9. 运维与风险

| 风险 | 说明 | 缓解 |
|------|------|------|
| cookie 过期 | 搜索强依赖登录态，会失效 | 扫码登录入口 + 失效检测提示重扫；抓取失败静默返回空不崩 |
| 无 GUI 环境扫码难 | 服务器跑扫码登录别扭 | 本机(有桌面)扫码存 `.xhs-auth.json` 后拷贝；或保持本地运行 |
| 被风控 | 频繁搜索可能限流 | 慢速滚动(已含随机延迟)、控制频率、单次数量适中 |
| 垃圾/广告帖 | 搜索结果混入课程/带货 | 移植脚本自带 24 词广告过滤 + AI 预筛 + 人工闸门 |
| 端口/启动 | 周边前端5173后端3000 | 见升级指南，勿撞画师库5174 |

## 10. 建议实施顺序（分阶段，可独立验收）

1. **P0 数据层**：建 `inspiration_candidate` + `crawl_run` 表（迁移脚本入 git）；先解决 `post_tags` 遗留隐患。
2. **P1 候选复核闭环**（不含爬虫，先跑通流程）：候选 CRUD + 复核转正/丢弃 + 前端复核页。**可先用「贴链接/手填」造几条候选验证转正链路**——此阶段零浏览器依赖。
3. **P2 关键词爬虫**：移植 Playwright 搜索脚本 + 扫码登录 + 发起采集接口，接进候选队列。
4. **P3 AI 预筛 + 批量**：接 AiAnalyzer 打分、按阈值批量操作。

> 这样即使 P2 的 cookie/浏览器出问题，P1 的候选复核闭环仍独立可用（手动贴链接也能进队列复核），降低整体风险。
