# 升级指南（给小小茶）

> 本指南由 xxwf 同步过来。请**你电脑上的 AI 助手**读取本文件，按顺序执行升级。
> 升级内容：同事近期对「周边可视化系统」的代码 + 数据库结构 + 数据迁移。
>
> **核心原则**：
> - **绝不**用同事的数据库文件替换你的库——那会冲掉你自己录入的数据。
> - 你的数据全程留在你自己的库里，只做「结构升级 + 你自己数据的迁移」。
> - 全程在你的项目目录、用你自己的 `.env` 配置执行。

---

## 前置确认（AI 先做）

1. 确认项目目录路径，所有命令在该目录下执行。
2. 读取你的 `backend/.env`，确认数据库连接配置（DB_HOST / DB_PORT / DB_NAME / DB_USER / DB_PASSWORD）。后续脚本会自动读这个文件，无需手动改。
3. 确认 MySQL 服务正在运行。
4. **强烈建议先备份你的数据库**（用 mysqldump 导出一份整库），万一升级出问题可还原：
   ```bash
   mysqldump -u<DB_USER> -p<DB_PASSWORD> <DB_NAME> > backup_升级前.sql
   ```

## 第三步：执行数据库结构升级（按版本顺序，逐个文件）

> ⚠️ **重要**：以下 SQL 每条**只能执行一次**。若报 "Duplicate column" / "Duplicate key" / "Can't DROP ... check column exists"，说明该条之前已执行过，**忽略即可**，继续下一条。
> 在 MySQL 里执行（命令行 / Navicat / 其它工具均可），连到你的 `.env` 里配的那个库。

**按这个顺序，逐个文件执行：**

### 3.1 `database/fix_schema_v14_play_count.sql`
作用：inspiration 表新增 `play_count`（播放量）字段。
```sql
ALTER TABLE inspiration ADD COLUMN play_count INT DEFAULT 0 AFTER view_count;
```

### 3.2 `database/fix_schema_v15_tag_index.sql`
作用：tag 表加两个索引，提升标签查询性能。
```sql
ALTER TABLE tag ADD INDEX idx_tag_type (tag_type);
ALTER TABLE tag ADD INDEX idx_status (status);
```

### 3.3 `database/fix_schema_v16_drop_dead_columns.sql`
作用：删除 3 个废弃字段（重复的/已迁移走的）。
```sql
ALTER TABLE project DROP COLUMN man_days;
ALTER TABLE project DROP COLUMN file_storage_path;
ALTER TABLE inspiration DROP COLUMN collect_count;
```
> 若报 "Can't DROP"：该列已删，忽略。

### 3.4 `database/fix_schema_v17_tag_relation.sql`
作用：新建标签关联表（inspiration_tag、project_tag），替代旧的逗号字符串存储。
```sql
CREATE TABLE IF NOT EXISTS inspiration_tag (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  inspiration_id BIGINT UNSIGNED NOT NULL,
  tag_type VARCHAR(20) NOT NULL COMMENT 'ip/category/craft/scene',
  tag_id BIGINT UNSIGNED NOT NULL,
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uk_insp_tag (inspiration_id, tag_type, tag_id),
  KEY idx_insp_type_tagid (tag_type, tag_id),
  KEY idx_insp_tagid (tag_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

CREATE TABLE IF NOT EXISTS project_tag (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  project_id BIGINT UNSIGNED NOT NULL,
  tag_type VARCHAR(20) NOT NULL COMMENT 'ip/category/craft/scene',
  tag_id BIGINT UNSIGNED NOT NULL,
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uk_proj_tag (project_id, tag_type, tag_id),
  KEY idx_proj_type_tagid (tag_type, tag_id),
  KEY idx_proj_tagid (tag_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```
> 这条用了 IF NOT EXISTS，重复执行安全。

**结构升级完成。** 验证：
```sql
SHOW COLUMNS FROM inspiration LIKE 'play_count';   -- 应有 1 行
SHOW INDEX FROM tag WHERE Key_name LIKE 'idx_tag%'; -- 应有 idx_tag_type、idx_status
SHOW TABLES LIKE '%_tag';                           -- 应有 inspiration_tag、project_tag
```

## 第四步：执行数据迁移脚本（按顺序）

> 这些脚本读你自己的 `.env`、迁移你自己的数据。**在 backend 目录下执行**。
> 每个脚本都支持 `--dry` 预览（只看不写），建议先 dry 看一眼再实跑。

```bash
cd backend
```

### 4.1 合并 collect_count → save_count
作用：把灵感表里冗余的 collect_count 收藏数迁移到 save_count（collect_count 字段已在上一步删除，若你库里 collect_count 本就为空可跳过）。
> ⚠️ 如果第三步已删 collect_count 列，此脚本会报错——**那说明你库里本来就没用这个字段，跳过 4.1 即可**。
```bash
node scripts/merge_collect_count.js --dry   # 预览
node scripts/merge_collect_count.js         # 实跑
```

### 4.2 迁移标签到关联表
作用：把 inspiration / project 里旧的逗号字符串标签（如 "1,5,63"）迁移到新建的关联表。
```bash
node scripts/migrate_tag_relation.js --dry  # 预览，看关联行数是否合理
node scripts/migrate_tag_relation.js        # 实跑
```
验证：应输出类似 "inspiration: N 条记录 → M 条关联行；project: N → M"。

### 4.3 把 project 的 IP 名称转成 id
作用：project 表的 ip_tag_ids 原本存的是名称（"无限暖暖"），统一转成 tag id（"3"）。
```bash
node scripts/migrate_project_ip_to_id.js --dry  # 预览
node scripts/migrate_project_ip_to_id.js        # 实跑
```
验证：输出 "转换 N，已是id跳过 M"，且校验无名称残留。

### 4.4 重算供应商统计字段
作用：从 project 表聚合每个供应商的合作项目数、总金额，回填到 supplier 表（之前一直是空的）。
```bash
node scripts/recalc_supplier_stats.js --dry  # 预览
node scripts/recalc_supplier_stats.js        # 实跑
```
验证：输出 "更新 N 个供应商"，如供应商有项目数应非 0。

## 第六步：验证升级成功

1. 打开系统，登录。
2. **灵感库**：列表能正常显示；按 IP/品类标签筛选，结果正确（不误匹配）。
3. **项目库**：列表 IP 列显示名称（不是数字 id）；新增/编辑项目时 IP 是下拉选标签；按 IP 筛选正常。
4. **供应商**：列表/看板的"合作项目数""合作总金额"有值（非全 0）。
5. **首页**：供应商排行有数据。
6. **删除**：灵感删除后右上角有 10 秒撤回按钮。
7. **回收站**：各模块（灵感/项目/价格/供应商/注意事项）都能进回收站、恢复、彻底删除。

任一项异常，对照本文档检查对应步骤是否漏跑/报错。

---

## 出问题怎么办

- **某条 SQL 报错**：先看是不是"已存在/已删除"类提示（正常，忽略）。其它错误把完整报错发给 xxwf。
- **某脚本报错**：先确认 `.env` 数据库配置正确、MySQL 在跑。仍报错把完整输出发给 xxwf。
- **升级后系统异常**：用第一步备份的 `backup_升级前.sql` 还原数据库 + `git reset` 回退代码，然后联系 xxwf。
  ```bash
  mysql -u<DB_USER> -p<DB_PASSWORD> <DB_NAME> < backup_升级前.sql
  ```

## 给执行此指南的 AI 助手

1. 严格按「第一步 → 第六步」顺序执行，不要跳步、不要打乱顺序。
2. 每一步执行后检查输出，确认成功再进下一步。
3. SQL 遇到 "Duplicate / Can't DROP / already exists" 类报错，判断为"已执行过"则忽略并继续，不要卡住。
4. 脚本先跑 `--dry` 预览，确认数字合理再实跑。
5. 执行前务必先备份数据库（前置确认第 4 点）。
6. 全程使用用户自己的 `.env` 配置，不要修改它。
7. 遇到本指南未覆盖的错误，停下来把完整报错反馈给用户，由用户联系 xxwf，不要自行猜测破坏性操作。
